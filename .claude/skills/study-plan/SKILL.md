# study-plan - 学习计划生成

生成代码主题的完整学习计划文档。

## 上下文

你正在辅助用户系统性地学习一个代码库的源码实现。
- 从 `.claude/memory/study-config.md` 读取源码路径和规范路径
- 产出物保存到 `study/plans/<主题>/`
- 生成多个 MD 文件：总览、数据结构、函数、调用链、规范对应

## 参数

```
/study-plan <topic>
```

- `topic`：主题名称，支持中文和空格
- 示例：`/study-plan rbuf 内存模型`、`/study-plan dds_write 写路径`

**参数处理规则**：
自动合并所有参数为完整主题名，无需引号。

## 执行步骤

### 1. 读取配置

使用 Read 工具读取 `.claude/memory/study-config.md`

获取：
- 源码路径
- 规范路径（可能为空）
- 源码模块映射
- 规范-源码映射

如果配置文件不存在：
```
⚠️ 还没有学习配置。请先运行 /study-start 初始化。
```
停止执行。

### 2. 分析主题

#### 2.1 定位代码元素

使用 Grep 在源码路径中搜索主题关键词：
- 搜索函数定义（output_mode: "content"，查看函数签名）
- 搜索结构体定义（搜索 `struct <name>` 或 `typedef struct`）
- 搜索类型定义

对于找到的每个元素，使用 Read 工具读取定义所在文件，提取：
- 精确的起始行号
- 结束行号（通过代码块分析）

#### 2.2 构建初步清单

收集以下信息并存储在内存中：
- 相关文件列表：`[{path, description}]`
- 相关结构体列表：`[{name, file, start_line, end_line}]`
- 相关函数列表：`[{name, file, start_line, end_line, signature}]`

#### 2.3 扩展分析

对于每个核心函数：
- 使用 Read 读取函数实现
- 分析函数内部调用的其他函数（识别 `func_name(` 模式）
- 识别函数使用的数据结构（识别变量声明和类型）
- 构建调用关系：`{caller: [callees]}`

对于每个核心结构体：
- 使用 Grep 查找使用该结构体的函数
- 识别结构体之间的包含关系（通过字段类型分析）

### 3. 范围确认（条件触发）

判断是否需要用户确认：
- 如果函数数量 > 15 个，或
- 如果文件数量 > 10 个

使用 AskUserQuestion 工具：

问题：「检测到以下相关内容，请选择要深入分析的部分」

选项（multiSelect: true）：
- 核心函数组 A：[列出 3-5 个代表函数名]
- 辅助函数组 B：[列出 3-5 个代表函数名]
- 数据结构组 C：[列出 3-5 个代表结构体名]
- 相关模块 D：[列出模块名]

根据用户选择过滤分析范围。

如果内容较少（≤15 个函数），跳过此步骤，全部分析。

### 4. 深度分析

#### 4.1 分析数据结构

对于每个结构体：
- 使用 Read 读取定义所在文件
- 提取结构体完整定义（起始行-结束行）
- 分析每个字段：名称、类型、用途（通过注释和命名推断）
- 使用 Grep 查找使用该结构体的函数
- 识别与其他结构体的关系（字段类型包含其他结构体）

构建结构体关系数据（用于生成 Mermaid classDiagram）

#### 4.2 分析函数

对于每个函数：
- 使用 Read 读取函数实现
- 提取函数签名、参数、返回值
- 分析关键逻辑（识别主要代码块、分支、循环）
- 识别函数内部调用的其他函数（搜索 `func_name(` 模式）
- 识别函数访问的数据结构（搜索结构体类型声明）
- 使用 Grep 查找调用该函数的位置

#### 4.3 构建调用链

基于函数调用关系：
- 识别入口函数（被外部调用的顶层函数）
- 构建多层调用关系（深度 3-5 层，避免过深）
- 标注数据流向（参数传递、返回值）
- 生成 Mermaid graph TD 数据

控制复杂度：
- 调用链节点数量 ≤ 20 个
- 对于大型函数（>500 行），只分析关键逻辑

#### 4.4 关联规范（可选）

如果 config 中有规范路径：
- 使用 Grep 在规范文件中搜索主题关键词
- 参考 config 中的「规范-源码映射」表
- 提取相关规范章节和描述
- 建立规范要求与源码实现的对应关系

### 5. 生成文档

#### 5.1 创建目录

主题名称处理：
- 去除特殊字符
- 空格替换为连字符
- 示例：「rbuf 内存模型」→ 「rbuf-内存模型」

使用 Bash 工具创建目录：
```bash
mkdir -p "study/plans/<处理后的主题名>"
```

计算从 `study/plans/<主题>/` 到项目根目录的相对路径前缀（通常为 `../../../`），用于所有源码链接。

#### 5.2 生成 00-overview.md

使用 Write 工具创建 `study/plans/<主题>/00-overview.md`

内容包含：
- 前置知识（从 study-config.md 的学习阶段推断）
- 架构概览（Mermaid graph TB，展示模块关系）
- 核心问题（3-5 个关键问题，**术语使用超链接跳转到 01-数据结构.md 对应章节**）
- 学习路径（链接到其他 MD 文件）
- 涉及文件表格（**文件名使用相对路径超链接**，而非纯文本）

**涉及文件表格格式**：
```markdown
| 文件 | 说明 |
|------|------|
| [file.c](../../../path/to/file.c) | 文件描述 |
```

**术语交叉链接格式**（用于核心问题等文本中的数据结构/函数名）：
```markdown
[rbufpool](./01-数据结构.md#1-struct-ddsi_rbufpool)
```
锚点基于目标文件中的标题自动生成（如 `## 1. \`struct xxx\`` → `#1-struct-xxx`）。

**超链接样式规则**：
- **链接文本不要用反引号包裹**，直接用普通文本。反引号会让链接在 Markdown 预览中显示为灰色代码样式，看不出是超链接
- ✅ 正确：`[file.c](../../../path/to/file.c)`
- ❌ 错误：`` [`file.c`](../../../path/to/file.c) ``
- 此规则适用于所有生成文档中的**所有链接**（涉及文件表格、源码位置、函数列表、调用链引用等）

#### 5.3 生成 01-数据结构.md

使用 Write 工具创建 `study/plans/<主题>/01-数据结构.md`

内容包含：
- 结构体关系图（Mermaid classDiagram）
- 逐个结构体详解：
  - 标题使用反引号：`## 1. \`struct xxx\``
  - 源码位置使用 blockquote + 相对路径：`> 📍 源码：[file.c](../../../相对路径/file.c) L起始-L结束`
  - 作用说明
  - **源代码定义**：必须包含结构体的原始 C 源代码（用 ```c 代码块），从源文件中读取并完整粘贴
  - 字段详解表格
  - 关键设计说明
  - **所有示意图使用 Mermaid 图（禁止使用 ASCII art / 文本框图）**

**源代码定义格式**（每个结构体必须包含）：
```markdown
### 源码定义

> 📍 源码：[ddsi_radmin.c](../../../cyclonedds/src/core/ddsi/src/ddsi_radmin.c#L264) L264-L288

` ` `c
struct ddsi_rbufpool {
  ddsrt_mutex_t lock;
  struct ddsi_rbuf *current;
  uint32_t rbuf_size;
  // ... 完整的结构体定义
};
` ` `
```

注意：使用 Read 工具读取源文件中结构体的**完整原始定义**，不要省略或重写。

#### 5.4 生成 02-核心函数.md

使用 Write 工具创建 `study/plans/<主题>/02-核心函数.md`

内容包含：
- 函数列表表格（**函数名和文件名必须使用相对路径超链接**）
  - 函数名链接到源文件对应行号：`[func_name](../../../path/to/file.c#L起始行)`
  - 文件名链接到源文件：`[file.c](../../../path/to/file.c)`
  - ✅ 正确示例：`| [ddsi_rmsg_new](../../../cyclonedds/src/core/ddsi/src/ddsi_radmin.c#L528) | [ddsi_radmin.c](../../../cyclonedds/src/core/ddsi/src/ddsi_radmin.c) | L528 | 分配新 rmsg |`
  - ❌ 错误示例：`| ddsi_rmsg_new | ddsi_radmin.c | L528 | 分配新 rmsg |`（纯文本，无法点击）
- 逐个函数详解：
  - 标题使用反引号：`### 1. \`func_name\``
  - 源码位置使用 blockquote + 相对路径：`> 📍 源码：[file.c](../../../相对路径/file.c#L起始行) L起始-L结束`
  - 函数签名代码块（必须用 ` ```c ` 标识语言）
  - 参数、返回值说明
  - 关键逻辑步骤
  - 调用的函数列表（带相对路径超链接）
  - 被调用位置列表（带相对路径超链接）

#### 5.5 生成 03-调用链.md

使用 Write 工具创建 `study/plans/<主题>/03-调用链.md`

内容包含：
- 完整调用链图（Mermaid graph TD）
- 数据流向图（Mermaid flowchart LR）
- 调用层次详解（按层级组织，**伪代码块必须使用 ` ```c ` 标注语言以启用语法高亮**）
- 关键路径分析（正常流程、错误处理）

**调用层次伪代码格式**：
```markdown
### 层级 1: 入口

` ` `c
// 使用 C 风格伪代码，确保语法高亮
recv_thread()
  while (running)
    do_packet(thrst, gv, conn, guidprefix, rbpool)
` ` `
```
（注意：上面的空格是为了避免嵌套代码块冲突，实际生成时不要加空格）

**禁止使用无语言标识的裸代码块**，否则渲染器不会高亮，用户阅读体验差。

#### 5.6 生成 04-规范对应.md（条件生成）

仅当 config 中有规范路径时：
使用 Write 工具创建 `study/plans/<主题>/04-规范对应.md`

内容包含：
- 规范章节列表
- 数据模型对照表
- 行为对照表
- 源码扩展功能
- 设计决策分析

## 格式规范（必须遵守）

### 链接格式

- **禁止使用 `file:///` 绝对路径**：大多数 Markdown 渲染器不支持，会显示为纯文本
- **必须使用相对路径**：从 `study/plans/<主题>/` 到项目根用 `../../../`
- **链接文本不要用反引号包裹**：反引号会让链接在 Markdown 预览中显示为灰色代码样式，看不出是超链接，用户无法识别为可点击链接
- **源码位置统一格式**：`> 📍 源码：[file.c](../../../相对路径/file.c#L起始行) L起始-L结束`
  - 超链接的 href 必须带 `#L起始行` 锚点（跳转到起始行），显示文本为 `L起始-L结束`（显示行范围）
  - ✅ 正确：`[ddsi_radmin.c](../../../cyclonedds/src/core/ddsi/src/ddsi_radmin.c#L264) L264-L288`
  - ❌ 错误：`[ddsi_radmin.c](../../../cyclonedds/src/core/ddsi/src/ddsi_radmin.c) L264-L288`（链接不带行号锚点）
- 文件名超链接格式：`[file.c](../../../相对路径/file.c)`
- 文档间交叉链接：`[术语名](./目标文件.md#锚点)`
- ✅ 正确示例：`[dds_init.c](../../../cyclonedds/src/core/ddsc/src/dds_init.c)`
- ❌ 错误示例：`` [`dds_init.c`](../../../cyclonedds/src/core/ddsc/src/dds_init.c) ``
- **所有出现源码文件名的地方都必须使用超链接**——包括但不限于：函数列表表格、源码位置标注、被调用位置、调用的函数列表、字段详解表格中的类型引用（如果引用了另一个已分析的结构体）。绝不允许把文件名写成纯文本

### 图表格式

- **所有图表必须使用 Mermaid**：禁止使用 ASCII art / 文本框图（┌──┐ 等字符画）
- **概念树、层级结构一律使用 `graph TD` 或 `mindmap`**：禁止使用 ``` 代码块内的 ASCII 树状图（`├──`, `└──` 等）
- 内存布局图使用 `graph LR` + `subgraph`
- 结构体关系用 `classDiagram`
- 调用链用 `graph TD`
- 数据流用 `flowchart LR`

**Mermaid 节点文本规范（防止渲染后看不清）**：
- **节点标签文本最多 25 个字符**（中文算 2 个字符宽度），超过则换行用 `<br/>`
- **多个 subgraph 横向排列时，使用 `graph LR`**，不要在 `graph TD` 中放太多并列的 subgraph
- **subgraph 标题简洁**，不超过 10 个中文字
- **禁止在节点标签中写完整的函数签名或长句**，只写函数名或关键词
- ✅ 正确：`S4["defrag 存储 rdata<br/>refcount += RDATA_BIAS"]`
- ❌ 错误：`S4["defrag 存储 rdata, refcount += RDATA_BIAS (2^20) * N（每个存储的片段）"]`（文本太长会被截断）
- **引用计数等含数学表达式的节点**：用 `<br/>` 换行，每行不超过 20 个字符宽度

### 表格图注

- 每个有标题意义的表格上方必须添加 blockquote 图注
- 格式：`> **图 N** 图注标题——简短描述`
- **表格中的路径、文件名必须使用相对路径超链接，而非纯文本反引号——无论在哪个文件中（包括 knowledge-map.md）**
- **链接文本不用反引号包裹**，确保在 Markdown 预览中显示为可点击的蓝色超链接
- **字段详解表格中引用其他已分析的结构体类型时，也要加超链接跳转到对应章节**
  - ✅ 正确：`| current | [ddsi_rbuf](./01-数据结构.md#4-struct-ddsi_rbuf)* | 当前活跃的 rbuf |`
  - ❌ 错误：`| current | ddsi_rbuf* | 当前活跃的 rbuf |`（纯文本，无法跳转）
- 示例：

```markdown
> **图 1** 源码模块映射——各模块与其源码目录的对应关系

| 模块 | 路径 | 说明 |
|------|------|------|
| DDSC | [cyclonedds/src/core/ddsc/](../../../cyclonedds/src/core/ddsc/) | DDS C API 实现层 |
```

### 文本编码

- 使用 UTF-8 编码写入所有文件
- 中文内容避免使用可能产生编码问题的特殊 Unicode 字符
- **禁止使用 box-drawing Unicode 字符**（`│`, `├`, `└`, `─`, `┌`, `┐`, `┘`, `┤` 等）生成树状图或概念图，这些字符在不同终端/编辑器中容易产生乱码或显示异常，一律使用 Mermaid 图替代
- **严格禁止生成任何乱码或不完整的 UTF-8 字符**：写入文件前确保所有中文字符完整无损，如果 Write 工具写入的内容出现截断（如"投递"变成"投�"），必须立即用 Read 检查并用 Edit 修复
- **写入文件后必须用 Read 工具抽查关键行**，确认没有乱码或截断字符

### 代码块语法高亮

- **所有代码块必须指定语言标识**，以确保语法高亮正确显示
- C 代码用 ` ```c `
- 伪代码或调用层次描述用 ` ```c ` 或 ` ```text `（选择最接近的语言）
- Shell 命令用 ` ```bash `
- Mermaid 图用 ` ```mermaid `
- **禁止使用无语言标识的裸代码块** ` ``` `，这会导致渲染器无法高亮，用户阅读体验差
- ✅ 正确：` ```c ` 后跟 C 代码
- ❌ 错误：` ``` ` 后跟 C 风格伪代码（无高亮）
